Phiên bản mở rộng giải thích chi tiết chữ "TẠI SAO" đằng sau mỗi quyết định kiến trúc, các giải pháp thay thế và sự đánh đổi (Trade-offs).

1 Khởi tạo Dự án & Lựa chọn Công cụ ORM

Khởi tạo Prisma:

Deep Dive: Tại sao chọn Prisma thay vì TypeORM hay Raw SQL?

1. Giải pháp thay thế 1: Raw SQL (Sử dụng thư viện 'pg' thuần)

Bạn tự viết các câu lệnh dạng SELECT * FROM users.

Ưu điểm: Hiệu năng cao nhất, dễ tối ưu query siêu phức tạp.
Trade-offs: Code rất dài, dễ bị lỗi SQL Injection nếu không cẩn thận, không có Type-safe (TypeScript không biết trong DB trả về cấu trúc gì).
2. Giải pháp thay thế 2: TypeORM hoặc Sequelize

Các ORM truyền thống áp dụng mô hình OOP (Class-based).

Ưu điểm: Phổ biến ở các dự án cũ, cấu trúc Class giống Java/C#.
Trade-offs: Dùng quá nhiều Decorators (@Entity, @Column) làm code rườm rà. Tính năng Type-safe không tuyệt đối.
3. Lựa chọn của chúng ta: Prisma ORM
Lý do chọn: Prisma sử dụng một file Schema độc lập (Declarative) rất dễ đọc. Nó tự động sinh ra mã (Auto-generated Client) giúp Type-safe đạt 100% (bạn gõ sai tên cột TypeScript sẽ báo lỗi ngay). Rất hoàn hảo cho Developer Experience (DX) cấp độ Junior/Mid.
Trade-offs: Prisma Core được viết bằng Rust, sinh ra một Engine chạy ngầm khá nặng. Trong môi trường Serverless (AWS Lambda) nó khởi động (Cold start) chậm hơn Raw SQL. Nhưng chạy trên VPS/Container (Docker) thì hoàn toàn tuyệt vời.

2 Cấu hình Cơ sở dữ liệu (Connection Pooling)

.env
# Chạy App (Pooler - Cổng 6543)
DATABASE_URL="postgresql://...pooler.supabase.com:6543/postgres?pgbouncer=true"

# Chạy Migrate (Direct - Cổng 5432)
DIRECT_URL="postgresql://...pooler.supabase.com:5432/postgres"

Deep Dive: Tại sao phải dùng 2 URL? Connection Pooling là gì?

Trong PostgreSQL, mỗi khi một User gọi API, Server Node.js phải tạo một "Kết nối" (Connection) tới DB. Việc tạo kết nối này cực kỳ tốn CPU và RAM. Postgres thông thường chỉ chịu được khoảng 100 kết nối đồng thời.

Giải pháp: Connection Pooling (PgBouncer)

Thay vì mở kết nối rồi đóng lại liên tục, ta đặt một "Người gác cổng" (PgBouncer) đứng ở cổng 6543. Người này duy trì sẵn một nhóm (Pool) khoảng 20-50 kết nối mở sẵn tới Postgres.

  • Khi App chạy (DATABASE_URL): Hàng nghìn request gọi vào API sẽ dùng cổng 6543. PgBouncer sẽ luân chuyển 50 kết nối mở sẵn này cho hàng nghìn request xài chung. Tốc độ cực nhanh.
  • Khi chạy Migrate (DIRECT_URL): Lệnh tạo bảng (Migration) đòi hỏi quyền kiểm soát độc quyền (Lock) để thay đổi cấu trúc DB. PgBouncer chia sẻ kết nối nên không cho phép làm điều này. Bắt buộc ta phải cấu hình DIRECT_URL nối thẳng vào cổng 5432 để Prisma xây bảng.

3 Thiết kế Schema (UUID & Phân quyền)

prisma/schema.prisma
model User {
  id                String         @id @default(uuid())
  username          String         @unique
  role              UserRole       @default(USER)
  // ...
  following         Followers[]    @relation("UserAsFollower")
  followers         Followers[]    @relation("UserAsFollowing")
}

model Followers {
  followerId  String
  followingId String
  status      FollowStatus @default(PENDING)
  @@id([followerId, followingId])
}

Deep Dive: Quyết định Kiến trúc Database

1. Tại sao dùng UUID thay vì ID tự tăng (1, 2, 3...)?
Giải pháp ID Int: Dễ đọc, DB tìm kiếm nhanh hơn.
Lý do chọn UUID: Trong kiến trúc Microservices, nếu bạn dùng ID tự tăng, Service A sinh ra User ID = 1, lỡ Service B cũng sinh ra dữ liệu ID = 1 sẽ bị đụng độ (Collision) khi gom dữ liệu. Ngoài ra, UUID (c3b9a2...) bảo mật hơn, đối thủ không thể cào (scrape) API /users/1, /users/2 để biết hệ thống của bạn có bao nhiêu user.
Trade-offs: UUID tốn 16 bytes lưu trữ (gấp 4 lần Int) và Index bị phân mảnh làm DB truy vấn chậm hơn một chút, nhưng với scale dự án này thì không đáng kể.
2. Tại sao khóa chính của Followers lại là @@id([followerId, followingId])?
Đây gọi là Composite Primary Key (Khóa chính tổ hợp). Nó đảm bảo cấp độ Database rằng: Cặp "A follow B" là DUY NHẤT. Dù Frontend có bị lag, User click nút Follow 10 lần cùng lúc (gây ra Race Condition), Postgres sẽ chặn đứng 9 request thừa ở tầng DB mà Backend không cần viết hàm kiểm tra dài dòng.

4 Middlewares: Phân tách Authentication và Authorization

Chúng ta chia làm 2 file Middleware riêng biệt.

src/middlewares/auth.middleware.ts
// Chỉ verify JWT Token
const decoded = jwt.verify(token, secret);
req.user = decoded; // { id, role }
src/middlewares/admin.middleware.ts
// Kiểm tra quyền
if (req.user?.role !== 'ADMIN') { return res.status(403)... }

Deep Dive: Authentication (401) vs Authorization (403)

1. Tại sao không gộp 2 cái này làm 1 file cho lẹ?

Áp dụng nguyên tắc Single Responsibility Principle (Đơn Trách Nhiệm).

  • auth.middleware giải quyết câu hỏi: "Bạn là ai?" (Authentication - Trả lỗi 401 Unauthorized). Hàm này dùng chung cho mọi API cần đăng nhập (đổi pass, nộp bài, xin follow).
  • admin.middleware giải quyết câu hỏi: "Tôi biết bạn là ai, nhưng bạn có quyền làm việc này không?" (Authorization - Trả lỗi 403 Forbidden). Hàm này chỉ gắn vào các API nhạy cảm (Xóa user, cấu hình hệ thống).
Trade-offs: Request đi qua nhiều tầng Middleware (Chain) sẽ làm mất thêm khoảng 1-2 mili-giây xử lý, bù lại code dễ bảo trì, tái sử dụng (Reusability) cao tuyệt đối.

5 Kiến trúc nhiều Tầng (Layered Architecture)

Bóc tách Database Query (Repository) ra khỏi Logic Nghiệp vụ (Service).

src/repositories/user.repository.ts
// Nơi DUY NHẤT gọi prisma.user...
async findByEmail(email: string) { return await prisma.user.findUnique(...) }
src/services/user.service.ts
// Nơi chứa não bộ: Hash password, check if user exists, sinh Token
const user = await this.userRepo.findByEmail(data.email);
if (!user) throw new Error(...)

Deep Dive: Tại sao phải dùng Repository Pattern?

Tại sao không gọi thẳng prisma.user.find() ở Controller cho nhanh?
Tình huống thực tế: Giả sử 1 năm sau, project quá lớn, CTO quyết định bỏ Prisma và chuyển sang dùng Raw SQL để tối ưu hiệu năng.

- Nếu bạn viết code thẳng vào Controller: Bạn phải mở hàng trăm file API ra, tìm từng dòng có chữ Prisma để sửa -> Dễ lỗi, ác mộng.
- Với Repository Pattern: Logic gọi Database bị nhốt lại trong file Repository. Bạn chỉ cần sửa 1 file Repository đó thay prisma.user bằng pg.query, còn tầng Service và Controller không hề biết và không cần thay đổi bất cứ dòng code nào. Đây gọi là tính chất Decoupling (Giảm kết dính).
