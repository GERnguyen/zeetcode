Ngày 1: Khởi Tạo UserService Với SQL (PostgreSQL & Prisma ORM) + Hệ Thống Follower

Ngày 1

Xây dựng một dịch vụ độc lập chịu trách nhiệm định danh, lưu trữ thông tin thực thể người dùng có cấu trúc quan hệ cao và thiết lập mô hình quan hệ tự liên kết (Self-Relation) để phục vụ tính năng Theo dõi (Follower/Following) xã hội.

1. Thiết kế Tầng Dữ Liệu (Prisma Schema)

Mô hình quan hệ giữa các lập trình viên sử dụng cấu trúc bảng trung gian quản lý hai liên kết đối xứng trỏ về cùng một bảng Người dùng:

// schema.prisma
datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

generator client {
  provider = "prisma-client-js"
}

enum PrivacySetting {
  PUBLIC
  FOLLOWERS_ONLY
  PRIVATE
}

model User {
  id               String         @id @default(uuid())
  username         String         @unique
  email            String         @unique
  passwordHash     String
  profileVisibility PrivacySetting @default(PUBLIC)
  createdAt        DateTime       @default(now())

  // Các mối quan hệ tự liên kết (Self-relations) qua bảng trung gian Follower
  following        Followers[]    @relation("UserAsFollower")
  followers        Followers[]    @relation("UserAsFollowing")
}

model Followers {
  followerId  String
  followingId String
  createdAt   DateTime @default(now())

  follower    User     @relation("UserAsFollower", fields: [followerId], references: [id], onDelete: Cascade)
  following   User     @relation("UserAsFollowing", fields: [followingId], references: [id], onDelete: Cascade)

  @@id([followerId, followingId]) // Khóa chính tổ hợp đảm bảo không thể follow trùng lặp
}
2. Triển Khai Luồng Nghiệp Vụ (Business Logic & Middleware)
  • Auth Middleware: UserService cấp mã JWT khi đăng nhập thành công. Các API Gateway hoặc Router cục bộ ở SubmissionService sẽ bóc tách mã Token này để điền giá trị req.user = { id: ... }.
  • API Quản Lý Mối Quan Hệ: Xây dựng các hàm điều khiển followUser(followerId, targetId) và unfollowUser(followerId, targetId) thực hiện trực tiếp thông qua Prisma Client.
📌 Điểm sáng Kiến Trúc trong CV: Chứng minh khả năng áp dụng nhuần nhuyễn mô hình CSDL Lai (Polyglot Persistence): Sử dụng thế mạnh toàn vẹn dữ liệu (ACID) tuyệt đối của RDBMS để xử lý tài khoản, các quan hệ ràng buộc khóa ngoại (Foreign Keys) phức tạp, đồng thời giữ nguyên MongoDB cho các log dữ liệu phi cấu trúc lớn.

Ngày 2: Triển Khai API Thống Kê & Thiết Lập Bộ Lọc Kiểm Tra Quyền Riêng Tư (Privacy Guard)

Ngày 2

Xây dựng thuật toán Aggregation tính toán tiến độ giải thuật tại SubmissionService kết hợp bộ lọc bảo mật phân quyền dữ liệu chéo giữa các Service dựa trên trạng thái quan hệ.

1. Thiết kế Luồng Giao Tiếp Liên Dịch Vụ (Inter-service Flow)
Client Yêu Cầu Stats User B -> SubmissionService -> Gửi HTTP Gọi Sang UserService -> Trả Về Trạng Thái Follow (True/False) + Privacy -> SubmissionService Tính Toán Hoặc Chặn (403 Forbidden)
2. Mã Nguồn Xử Lý Bảo Mật Quyền Riêng Tư (Submission Controller)
import { Request, Response } from 'express';
import axios from 'axios';
import { SubmissionRepository } from '../repositories/submission.repository';

const submissionRepo = new SubmissionRepository();

export const getCompletionStats = async (req: Request, res: Response) => {
    const requestorId = req.user.id; // Lấy từ mã JWT Token người gửi request
    const targetUserId = req.params.userId; // Người dùng cần xem hồ sơ

    if (requestorId === targetUserId) {
        // Nếu tự xem chính mình, bỏ qua kiểm tra quyền bảo mật
        const stats = await submissionRepo.aggregateUserStats(targetUserId);
        return res.status(200).json({ success: true, data: stats });
    }

    try {
        // Thực hiện lệnh gọi HTTP nội bộ sang UserService để xác thực mối quan hệ xã hội
        const userCheckResponse = await axios.get(`http://localhost:4000/api/v1/users/${targetUserId}/privacy-check`, {
            params: { viewerId: requestorId }
        });

        const { isAllowed } = userCheckResponse.data;
        if (!isAllowed) {
            return res.status(403).json({ 
                success: false, 
                message: "Hồ sơ của người dùng này được đặt ở chế độ riêng tư hoặc chỉ hiển thị với người theo dõi!" 
            });
        }

        const stats = await submissionRepo.aggregateUserStats(targetUserId);
        return res.status(200).json({ success: true, data: stats });

    } catch (error) {
        return res.status(500).json({ success: false, message: "Lỗi kết nối liên dịch vụ giữa các microservices." });
    }
};
3. Thuật Toán Thống Kê Tiến Độ Với MongoDB Aggregation Pipeline
// submission.repository.ts
async aggregateUserStats(userId: string) {
    return await SubmissionModel.aggregate([
        { $match: { userId: userId, status: 'Accepted' } },
        { 
            $group: { 
                _id: "$difficulty", 
                uniqueSolvedProblems: { $addToSet: "$problemId" } // Lọc bỏ trùng lặp nếu nộp 1 bài nhiều lần
            } 
        },
        {
            $project: {
                difficulty: "$_id",
                count: { $size: "$uniqueSolvedProblems" },
                _id: 0
            }
        }
    ]);
}

Ngày 3: Hoàn Thiện Module Bài Giải Nâng Cao (Editorial) & Ánh Xạ Đường Dẫn Băng Thông S3/CDN

Ngày 3

Tăng cường sức mạnh nội dung cho ProblemService bằng cách tích hợp trường dữ liệu Hướng Dẫn Giải chi tiết, kết hợp cấu trúc DTO linh hoạt xử lý luồng Video tùy biến theo hạ tầng phát triển.

1. Mở Rộng Cấu Trúc Dữ Liệu Bài Tập (Mongoose Schema)
// problem.model.ts
import { Schema, model } from 'mongoose';

const problemSchema = new Schema({
    title: { type: String, required: true, trim: true },
    description: { type: String, required: true },
    difficulty: { type: String, enum: ['easy', 'medium', 'hard'], required: true },
    testCases: [{ input: String, output: String }],
    editorial: {
        solutionMarkdown: { type: String, default: "" },
        videoProvider: { type: String, enum: ['youtube', 'cdn'], default: 'youtube' },
        videoKey: { type: String, default: "" } // Lưu ID Youtube hoặc Object Key trên AWS S3
    }
}, { timestamps: true });

export const Problem = model('Problem', problemSchema);
2. Giải Pháp Ánh Xạ DTO Động Theo Môi Trường Triển Khai

Nhằm tối ưu hóa băng thông và chi phí cho từng giai đoạn, tầng DTO (Data Transfer Object) sẽ thực hiện nhiệm vụ biên dịch dữ liệu trước khi trả về phía Client:

// problem.dto.ts
export class ProblemResponseDTO {
    id: string;
    title: string;
    description: string;
    difficulty: string;
    editorial: {
        markdown: string;
        videoUrl: string;
    };

    constructor(problemDoc: any) {
        this.id = problemDoc._id;
        this.title = problemDoc.title;
        this.description = problemDoc.description;
        this.difficulty = problemDoc.difficulty;
        
        let compiledVideoUrl = "";
        if (problemDoc.editorial?.videoProvider === 'youtube') {
            compiledVideoUrl = `https://www.youtube.com/embed/${problemDoc.editorial.videoKey}`;
        } else if (problemDoc.editorial?.videoProvider === 'cdn') {
            // Tự động cấu hình ánh xạ sang domain của AWS CloudFront CDN khi chạy Production
            const cdnDomain = process.env.AWS_CLOUDFRONT_DOMAIN || "https://static.zeetcode.io";
            compiledVideoUrl = `${cdnDomain}/videos/${problemDoc.editorial.videoKey}`;
        }

        this.editorial = {
            markdown: problemDoc.editorial?.solutionMarkdown || "",
            videoUrl: compiledVideoUrl
        };
    }
}

Ngày 4: Xây Dựng Bản Đồ Học Tập (Roadmap Skill Tree) Bằng Mô Hình Đồ Thị Cấu Trúc Dữ Liệu

Ngày 4

Thiết kế hệ thống sơ đồ cây lộ trình phân cấp bằng cấu trúc danh sách kề (Adjacency List Model) trong NoSQL để quản lý các node phụ thuộc học thuật đan xen phức tạp một cách gọn gàng.

1. Định Nghĩa Thực Thể Đồ Thị Định Hướng (Directed Graph Schema)
// roadmapNode.model.ts
import { Schema, model } from 'mongoose';

const roadmapNodeSchema = new Schema({
    name: { type: String, required: true, unique: true }, // "Arrays & Hashing"
    slug: { type: String, required: true, unique: true, index: true }, // "arrays-hashing"
    description: { type: String, default: "" },
    
    // Mảng chứa slug của tất cả các Node bắt buộc phải hoàn thành trước đó (Khóa liên kết Đồ thị)
    parentNodes: [{ type: String }], 
    
    // Liên kết tập hợp các bài tập bắt buộc thuộc nhóm chủ đề này
    problems: [{ type: Schema.Types.ObjectId, ref: 'Problem' }] 
}, { timestamps: true });

export const RoadmapNode = model('RoadmapNode', roadmapNodeSchema);
2. Tầng Controller Truy Vấn Trả Về Cấu Trúc Đồ Thị Tuyến Tính
// roadmap.controller.ts
import { Request, Response } from 'express';
import { RoadmapNode } from '../models/roadmapNode.model';

export const getFullRoadmap = async (req: Request, res: Response) => {
    // Thực hiện truy vấn đồng thời nạp đầy đủ thông tin đề bài (populate) từ bảng Problem liên kết
    const fullGraph = await RoadmapNode.find({})
        .populate('problems', 'title difficulty')
        .lean();

    return res.status(200).json({
        success: true,
        message: "Tải bản đồ lộ trình cấu trúc đồ thị thành công!",
        data: fullGraph
    });
};
💡 Tư duy Thuật toán thực tế: Mô hình này mô phỏng hoàn hảo một **Đồ thị có hướng không chu trình (DAG - Directed Acyclic Graph)**. Tại Frontend, cấu trúc mảng danh sách kề này sẽ được truyền trực tiếp vào các thư viện vẽ đồ thị chuyên sâu như React Flow để tính toán tọa độ vẽ đường nối mũi tên liên kết giữa các bài học.

2. Cơ Chế Đánh Chặn Hoạt Động (Contest Validation Hook)

Tại API nộp bài của SubmissionService, hệ thống cần bổ sung một hàm Hook trung gian để kiểm tra xem lượt submit code này có thuộc về một cuộc thi đang diễn ra hay không:

// submission.service.ts Luồng xử lý tạo Submission
async createSubmission(data: any) {
    const { userId, problemId, code, language, contestId } = data;

    let isContestValid = false;
    if (contestId) {
        // Thực hiện lệnh gọi chéo hoặc truy vấn thông tin thời gian của giải đấu
        const contest = await ContestModel.findById(contestId);
        const now = new Date();
        
        if (contest && now >= contest.startTime && now <= contest.endTime) {
            isContestValid = true;
        }
    }

    const submission = await SubmissionModel.create({
        userId,
        problemId,
        code,
        language,
        difficulty: data.difficulty,
        // Nếu hợp lệ, gắn nhãn contestId vào bản ghi để phục vụ tính điểm xếp hạng ở Ngày 6
        contestId: isContestValid ? contestId : null 
    });

    // Đẩy tác vụ vào hàng đợi Message Queue để chấm điểm bất đồng bộ
    await submissionProducer.pushToQueue(submission);
    return submission;
}

Ngày 6: Xây Dựng Bảng Xếp Hạng Siêu Tốc Thời Gian Thực Với Redis Sorted Sets (ZSET)

Ngày 6

Giải quyết triệt để bài toán thắt nút cổ chai (Bottleneck) hiệu năng khi hàng nghìn thí sinh cùng submit code và tra cứu bảng xếp hạng liên tục bằng cách ứng dụng cấu trúc cấu trúc dữ liệu Sorted Sets của Redis.

1. Kiến Trúc Luồng Cập Nhật Điểm Bất Đồng Bộ
EvaluationWorker (Docker chạy xong bài) -> Kết Quả: Accepted -> Gọi Hàm Cộng Điểm Vào Redis ZSET -> API Contest Đọc Trực Tiếp Từ RAM Redis Siêu Tốc
2. Thuật Toán Cập Nhật Điểm Và Giải Quyết Tranh Chấp Trùng Điểm (Tie-Breaking)

Trong thi đấu giải thuật, nếu hai người có cùng điểm số, người nào hoàn thành cuộc thi với **tổng thời gian sớm hơn** sẽ đứng trước. Chúng ta áp dụng thuật toán ép giá trị thời gian (Timestamp) vào phần thập phân của điểm số:

// leaderboard.service.ts (Chạy bên trong Evaluation Service Worker)
import { redisClient } from '../config/redis.config';

export const processContestLeaderboard = async (submission: any) => {
    if (!submission.contestId || submission.status !== 'Accepted') return;

    const { contestId, userId } = submission;
    const baseScoreIncrement = 100; // Mỗi bài toán giải đúng được cộng 100 điểm gốc

    // Thuật toán Tie-Breaking: Lấy khoảng cách thời gian còn lại tối đa của vũ trụ trừ đi thời gian hiện tại
    // Giúp cho người submit sớm hơn sẽ có phần thập phân LỚN HƠN, từ đó xếp hạng CAO HƠN khi sắp xếp giảm dần
    const maxFutureTimestamp = 9999999999;
    const currentTimestamp = Math.floor(Date.now() / 1000);
    const tieBreaker = (maxFutureTimestamp - currentTimestamp) / maxFutureTimestamp;

    // Ví dụ điểm thực tế ghi vào Redis: 100.83748293
    const finalScoreToIncrement = baseScoreIncrement + tieBreaker;

    // Sử dụng lệnh ZINCRBY của Redis để cộng dồn điểm số nguyên tử (Atomic) trong bộ nhớ RAM
    await redisClient.zincrby(`contest:leaderboard:${contestId}`, finalScoreToIncrement, userId);
};
3. API Lấy Bảng Xếp Hạng Tốc Độ Cao (Contest Controller)
export const getLeaderboard = async (req: Request, res: Response) => {
    const { contestId } = req.params;
    
    // Truy vấn nhanh phạm vi từ vị trí 0 đến 9 để lấy Top 10 kèm điểm số từ Redis RAM (Thời gian phản hồi < 2ms)
    const rawLeaderboard = await redisClient.zrevrange(`contest:leaderboard:${contestId}`, 0, 9, 'WITHSCORES');
    
    // Định dạng lại mảng thô [user_id_1, score_1, user_id_2, score_2] thành JSON hoàn chỉnh
    const formattedLeaderboard = [];
    for(let i = 0; i < rawLeaderboard.length; i += 2) {
        formattedLeaderboard.push({
            rank: (i / 2) + 1,
            userId: rawLeaderboard[i],
            totalScore: Math.floor(parseFloat(rawLeaderboard[i+1])) // Lấy phần nguyên hiển thị cho người dùng
        });
    }

    return res.status(200).json({ success: true, data: formattedLeaderboard });
};

Ngày 7: Gia Cố Phòng Thủ Hệ Thống (Rate Limiting Middleware) & Giới Hạn Tài Nguyên Docker Container

Ngày 7

Ngày cuối cùng tập trung hoàn toàn vào việc tối ưu an toàn hệ thống, phòng chống tấn công từ chối dịch vụ (DDoS Spam Submit) và cấu hình cô lập tài nguyên phần cứng cho môi trường Sandbox.

1. Triển Khai Middleware Chống Spam Code Sử Dụng Thuật Toán Token Bucket Tinh Gọn
// rateLimiter.middleware.ts
import { Request, Response, NextFunction } from 'express';
import { redisClient } from '../config/redis.config';

export const submissionRateLimiter = async (req: Request, res: Response, next: NextFunction) => {
    const userId = req.user?.id || req.ip; // Định danh dựa trên ID tài khoản hoặc IP mạng
    const redisKey = `limiter:submission:${userId}`;

    // Tăng giá trị đếm nguyên tử trong Redis
    const requestCount = await redisClient.incr(redisKey);
    
    if (requestCount === 1) {
        // Đặt thời gian đóng băng hết hạn là 15 giây. Trong 15 giây này không được submit quá 1 lần.
        await redisClient.expire(redisKey, 15);
    }

    if (requestCount > 1) {
        return res.status(429).json({
            success: false,
            message: "Hệ thống phát hiện hành vi spam! Vui lòng đợi 15 giây giữa các lần nộp mã nguồn."
        });
    }

    next();
};
2. Giới Hạn Phần Cứng Sandbox Chống Vòng Lặp Vô Hạn (createContainer.util.ts)

Nếu người dùng nộp một đoạn code có chứa mã độc hoặc vòng lặp vô hạn (while(true)), nó có thể ăn hết CPU/RAM của server host. Chúng ta cần ép Docker giới hạn nghiêm ngặt bộ nhớ phần cứng cứng:

// Cấu hình tích hợp bổ sung vào hàm createContainer sử dụng dockerode
const container = await docker.createContainer({
    Image: image,
    Cmd: bootCmd,
    AttachStdout: true,
    AttachStderr: true,
    HostConfig: {
        Memory: 128 * 1024 * 1024, // Giới hạn cứng tối đa 128MB RAM cho mỗi container chạy code
        MemorySwap: 0,             // Tắt hoàn toàn bộ nhớ Swap ảo để bảo vệ ổ đĩa cứng
        CpuQuota: 50000,           // Giới hạn thời gian sử dụng CPU tối đa 50% của 1 Core vật lý
        CpuPeriod: 100000,
        NetworkMode: 'none'        // Cực kỳ quan trọng: Tắt toàn bộ kết nối Internet của container sandbox để chống rò rỉ dữ liệu
    }
});
🛡️ Đánh giá từ Chuyên Gia Phỏng Vấn Kỹ Thuật: Việc biết ngắt kết nối mạng (NetworkMode: 'none') và khống chế RAM/CPU vật lý của Container Sandbox chứng minh ứng viên có tư duy cực kỳ sâu sắc về **An toàn bảo mật hệ thống (Infra Security)**, hiểu rõ rủi ro khi thực thi mã nguồn lạ (Untrusted Code Execution) trên máy chủ doanh nghiệp.
