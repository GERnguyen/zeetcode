Zeetcode Deploy Guide: Free-Tier Demo

Tài liệu này hướng dẫn deploy Zeetcode để có link demo đưa vào CV. Giả định hiện tại: bạn đã có MongoDB online, đã có Supabase Postgres online, và muốn chạy backend/judge miễn phí hoặc gần miễn phí nhất có thể.

Frontend Deploy bằng Vercel/Netlify/Cloudflare Pages hoặc serve static trên VM.
Backend Chạy 5 Node services trên một VM có Docker, Redis và PM2.
Judge Docker sandbox chạy trên VM. Không nên deploy EvaluationService lên serverless.

1. Trạng thái code sau chỉnh sửa

  • Docker judge không còn hard-code 1GB. Memory lấy từ JUDGE_MEMORY_MB, mặc định 256MB.
  • BullMQ worker đã có JUDGE_WORKER_CONCURRENCY, mặc định 1. Với free tier, giữ concurrency 1 để tránh nhiều container chạy cùng lúc làm sập máy.
  • .gitignore đã ignore .env thật ở mọi service, nhưng vẫn cho commit .env.example.
  • Các service Mongo đã có retry boot, heartbeat, pool size nhỏ, và log reconnect rõ hơn.

2. Nên deploy theo mô hình nào?

Vì Zeetcode có Docker judge, lựa chọn free hợp lý nhất là một VM luôn chạy được Docker. Với yêu cầu miễn phí, hướng thực tế nhất là Oracle Cloud Always Free VM. Oracle docs liệt kê Always Free resources cho Compute, trong đó Ampere A1 là lựa chọn đáng cân nhắc cho app nhiều service.

Thành phần Khuyến nghị free-tier Lý do
Frontend Vercel hoặc serve bằng Nginx trên VM Frontend chỉ là static build, không tốn tài nguyên nhiều.
UserService Oracle VM + Supabase Postgres Service Node chạy ổn trên VM; DB đã managed.
Problem/Submission/Battle Oracle VM + MongoDB Atlas App giữ connection tới Mongo online; pool nhỏ để không quá quota.
Redis Self-host Redis trên VM BullMQ và matchmaking dùng Redis liên tục; self-host đơn giản và ổn hơn free serverless Redis.
EvaluationService Oracle VM có Docker daemon Judge cần tạo container runtime; serverless/PaaS thường không cho quyền này.

Nếu bạn chưa có domain, cách miễn phí nhất là serve cả frontend và backend qua http://PUBLIC_IP. Link CV nhìn không đẹp bằng domain HTTPS, nhưng demo vẫn chạy. Nếu dùng Vercel HTTPS cho frontend, backend cũng nên có HTTPS để tránh mixed content.

3. Tạo lại toàn bộ secrets trước deploy

Không dùng lại secrets local cũ. Trước khi public, tạo token mới bằng lệnh sau trên máy local hoặc trên server: 

openssl rand -hex 32

Tạo ít nhất các secret sau:

Tên logic Dùng cho env nào Ghi chú
JWT_ACCESS_SECRET UserService, ProblemService, SubmissionService, BattleService Cùng một giá trị để các service verify access token giống nhau.
JWT_REFRESH_SECRET UserService Chỉ UserService dùng để issue/verify refresh token.
PROBLEM_INTERNAL_TOKEN ProblemService INTERNAL + Submission/Battle PROBLEM token Cho phép service nội bộ đọc problem/testcase nhạy cảm.
SUBMISSION_INTERNAL_TOKEN SubmissionService INTERNAL + Evaluation SUBMISSION token Cho phép evaluator cập nhật status submission.
USER_INTERNAL_TOKEN UserService INTERNAL + Battle USER token Cho phép BattleService đọc user profile/elo nội bộ.
Email app password UserService MAIL_PASS Dùng app password, không dùng mật khẩu chính của email.
DB URLs DB_URL, DATABASE_URL, DIRECT_URL Lấy lại từ MongoDB Atlas và Supabase dashboard.

4. Mapping service token đúng cặp

Đây là phần dễ sai nhất. Hãy nghĩ theo cặp: service A có INTERNAL_SERVICE_TOKEN, service B muốn gọi A thì B phải gửi token đó.

Token gốc Đặt ở service chủ Đặt ở service gọi
PROBLEM_INTERNAL_TOKEN ProblemService.INTERNAL_SERVICE_TOKEN SubmissionService.PROBLEM_SERVICE_TOKEN
BattleService.PROBLEM_SERVICE_TOKEN
SUBMISSION_INTERNAL_TOKEN SubmissionService.INTERNAL_SERVICE_TOKEN EvaluationService.SUBMISSION_SERVICE_TOKEN
USER_INTERNAL_TOKEN UserService.INTERNAL_SERVICE_TOKEN BattleService.USER_SERVICE_TOKEN

5. Chuẩn bị VM backend

1

Tạo VM

Tạo Ubuntu VM trên Oracle Cloud Always Free. Nên chọn Ampere A1 nếu tenancy còn capacity. Nếu chỉ lấy được VM 1GB RAM thì app vẫn có thể chạy demo, nhưng Docker judge sẽ rất chật; lúc đó giữ JUDGE_MEMORY_MB=256 và concurrency 1.

2

Mở port

Mở inbound port trong Oracle Security List:

  • 22 cho SSH.
  • 80 cho HTTP demo.
  • 443 nếu có HTTPS/domain.
3

Cài runtime

sudo apt update
sudo apt install -y git curl nginx redis-server

curl -fsSL https://deb.nodesource.com/setup_22.x | sudo -E bash -
sudo apt install -y nodejs

sudo npm install -g pm2

curl -fsSL https://get.docker.com | sudo sh
sudo usermod -aG docker $USER

Logout rồi SSH lại để quyền Docker group có hiệu lực.

4

Pull Docker images cho judge

docker pull python:3.8-slim
docker pull gcc:latest

7. Build frontend

Nếu serve frontend trực tiếp bằng VM/Nginx, tạo file fe/.env.production:

VITE_USER_API=http://YOUR_VM_PUBLIC_IP/user-api/api/v1
VITE_PROBLEM_API=http://YOUR_VM_PUBLIC_IP/problem-api/api/v1
VITE_SUBMISSION_API=http://YOUR_VM_PUBLIC_IP/submission-api/api/v1
VITE_BATTLE_API=http://YOUR_VM_PUBLIC_IP/battle-api/api/v1
VITE_BATTLE_SOCKET=http://YOUR_VM_PUBLIC_IP
cd fe
npm ci
npm run build
cd ..

Nếu deploy frontend bằng Vercel, đặt các biến VITE_* tương tự trong Vercel Project Settings. Vercel docs có phần Environment Variables riêng cho cách cấu hình này.

9. Nginx reverse proxy

Tạo file /etc/nginx/sites-available/zeetcode:

server {
  listen 80;
  server_name _;

  root /home/ubuntu/zeetcode/fe/dist;
  index index.html;

  location / {
    try_files $uri $uri/ /index.html;
  }

  location /user-api/ {
    proxy_pass http://127.0.0.1:3003/;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
  }

  location /problem-api/ {
    proxy_pass http://127.0.0.1:3000/;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
  }

  location /submission-api/ {
    proxy_pass http://127.0.0.1:3001/;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
  }

  location /battle-api/ {
    proxy_pass http://127.0.0.1:3004/;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
  }

  location /socket.io/ {
    proxy_pass http://127.0.0.1:3004/socket.io/;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
    proxy_set_header Host $host;
    proxy_read_timeout 75s;
  }
}
sudo ln -s /etc/nginx/sites-available/zeetcode /etc/nginx/sites-enabled/zeetcode
sudo nginx -t
sudo systemctl reload nginx

10. Database checklist

MongoDB Atlas

  • Tạo database user riêng cho app, không dùng admin/root.
  • Network Access: thêm public IP của VM. Nếu cần demo nhanh có thể dùng 0.0.0.0/0, nhưng production nên giới hạn IP.
  • Dùng cùng cluster nhưng tách database theo service nếu muốn dễ quản lý.

Supabase

  • Lấy connection string từ Dashboard > Connect.
  • Với app server lâu dài, direct connection thường đơn giản. Nếu dùng pooler, đảm bảo Prisma config tương thích.
  • Chạy Prisma migration trước khi public nếu schema chưa có trên Supabase.
cd /home/ubuntu/zeetcode/be/UserService
npx prisma migrate deploy

11. Smoke test sau deploy

  1. Mở http://YOUR_VM_PUBLIC_IP, trang frontend phải load được.
  2. Register tài khoản mới, check email verify.
  3. Login, vào Practice, mở problem.
  4. Run sample Python, sau đó submit Python.
  5. Chuyển sang C++, submit lại để kiểm tra Docker image gcc.
  6. Mở 2 browser/user khác nhau, test ranked battle matching.
  7. Tạo private room, join bằng room ID, start match.
  8. Kiểm tra PM2 logs không có lỗi Mongo disconnect kéo dài.
pm2 logs zeetcode-problem-service
pm2 logs zeetcode-submission-service
pm2 logs zeetcode-battle-service
pm2 logs zeetcode-evaluation-service

12. Bật CI/CD deploy tự động

Repo đã có workflow CI/CD. Ban đầu để deploy tắt. Khi server đã chạy ổn, vào GitHub: Settings -> Secrets and variables -> Actions.

Repository variable

ENABLE_DEPLOY=true

Repository secrets

DEPLOY_HOST=YOUR_VM_PUBLIC_IP
DEPLOY_USER=ubuntu
DEPLOY_PORT=22
DEPLOY_PATH=/home/ubuntu/zeetcode
DEPLOY_SSH_KEY=YOUR_PRIVATE_SSH_KEY_FOR_DEPLOY

Sau đó mỗi lần merge vào main, CI chạy trước. Nếu CI pass, workflow Deploy sẽ SSH vào server, pull code mới nhất, typecheck backend, rồi restart PM2 services.

13. Khi nào tăng judge memory?

Giá trị Khi dùng Tradeoff
JUDGE_MEMORY_MB=256 Demo CV, bài algorithm nhỏ, VM ít RAM. An toàn tài nguyên nhất, nhưng bài C++/Python nặng có thể fail memory.
JUDGE_MEMORY_MB=512 VM có từ 2GB RAM trở lên. Ít false fail hơn, nhưng mỗi container tốn nhiều RAM hơn.
JUDGE_MEMORY_MB=1024 Chỉ nên dùng local hoặc server mạnh. Không phù hợp free tier nếu có nhiều submission.

Với free deploy, giữ JUDGE_WORKER_CONCURRENCY=1. Nếu nhiều người submit cùng lúc, queue chậm nhưng server vẫn sống.
