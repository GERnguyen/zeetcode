01 · Executive summary

Dự án đang giải quyết bài toán gì?

Zeetcode là một nền tảng luyện coding có hai nhánh sản phẩm: luyện tập cá nhân giống LeetCode và đấu realtime theo runtime. Dự án không chỉ là CRUD bài tập; nó có judge bất đồng bộ, xác thực người dùng, email verification, lịch sử submission, battle ranked/private, ELO và realtime event.

Mức độ CV: mạnh

Practice

Người dùng xem danh sách problem, filter/sort/search, đọc description/editorial, viết code Python/C++, run sample, submit full test, xem history và dùng lại code cũ.

Judge engine

Submission được lưu ở database, đẩy vào Redis/BullMQ, EvaluationService chạy code trong Docker container, sau đó callback kết quả về SubmissionService bằng service token.

Battle arena

Ranked matchmaking dựa trên ELO range, private room bằng room code, realtime socket cho queue, room status, timer, submission event, leave/forfeit và kết quả trận đấu.

Luận điểm chính khi đưa vào CV: Đây là một full-stack distributed system nhỏ với nhiều concern thực tế: authentication, asynchronous job processing, sandboxed code execution, realtime coordination, domain-specific ranking, data modeling và UX cho editor.

Tư duy thiết kế nên nhớ

Không chỉ “làm chạy”

Với mỗi phần, câu hỏi đúng là: dữ liệu này thuộc ownership của service nào, request có thể chậm/lỗi không, user có cần realtime không, và nếu scale lên thì bottleneck nằm ở đâu.

Cách thay thế luôn tồn tại

Ví dụ: judge có thể chạy sync trong API, battle có thể polling thay vì socket, toàn backend có thể monolith thay vì microservice. Report này nêu rõ tại sao chọn cách hiện tại.

Tradeoff là thứ cần nói được

Một lựa chọn tốt trong demo/CV chưa chắc tốt nhất cho production. Quan trọng là hiểu chi phí: độ phức tạp, consistency, vận hành, bảo mật và khả năng debug.

Service boundaries

  • UserService: user identity, JWT, refresh token rotation, email verification, profile, ELO storage.
  • ProblemService: problem catalog, markdown sanitation, public problem view, judge-only testcase payload.
  • SubmissionService: submission lifecycle, practice run vs real submit, accepted stats, queue producer.
  • EvaluationService: queue worker, Docker execution, verdict calculation, callback to SubmissionService.
  • BattleService: ranked queue, private rooms, Socket.IO events, battle lifecycle, result and ELO update.

Runtime infrastructure

  • MongoDB dùng cho problem, submission, battle room vì schema có document tự nhiên và thay đổi nhanh.
  • PostgreSQL dùng cho user/auth vì cần constraint unique, relation token và transaction.
  • Redis dùng cho BullMQ submission queue và sorted set matchmaking queue.
  • Docker dùng làm sandbox chạy code Python/C++ theo từng testcase.

03 · Frontend architecture

Frontend: SPA có editor, realtime và state sync

Frontend nằm trong fe/src, dùng route-based feature organization. Các page chính là auth, practice, battle, history và profile.

Routing

AppRouter tách route public email action khỏi authenticated app. Nếu chưa có access token thì render AuthScreen; nếu đã đăng nhập thì render AppShell với nested routes.

Server state

React Query quản lý problem list, problem detail, submission history, accepted stats, battle history và current battle state. Các mutation invalidate query phù hợp.

Client state

Zustand lưu access token, refresh token và user trong localStorage/store. Socket connection được quản lý ở từng battle page bằng ref và effect cleanup.

04 · Backend services

Phân tích từng service

Mỗi service là một Express app TypeScript riêng. Các service chia sẻ pattern: config từ env, correlation id, logger Winston, Zod validation và AppError middleware.

UserService

Service quản lý identity. Password hash bằng bcrypt. Access token và refresh token tách secret, refresh token được lưu dạng SHA-256 hash trong PostgreSQL và rotate khi refresh. EmailToken được dùng cho signup confirm, reset password và sensitive password change.

PostgreSQL Prisma bcryptjs JWT Nodemailer

ProblemService

Service lưu problem, testcase, editorial và metadata. Public endpoint chỉ trả examples 2 testcase đầu, còn full testcase chỉ mở qua /problems/:id/judge bằng service token. Markdown description/editorial được sanitize khi tạo/cập nhật.

MongoDB Mongoose sanitize-html marked

SubmissionService

Service nhận code từ user, validate access token, fetch problem judge payload từ ProblemService, lưu submission với status QUEUED, rồi enqueue BullMQ job. Nó cũng cung cấp accepted stats và submission history theo user/problem.

MongoDB BullMQ producer Service token

EvaluationService

Service worker consume BullMQ. Với mỗi job, nó set submission RUNNING, chạy từng testcase tuần tự trong Docker, tổng hợp verdict, runtime, passed/total, rồi callback PATCH /submissions/:id/status.

BullMQ worker Dockerode Python/C++

BattleService

Service quản lý ranked queue bằng Redis sorted set, room state trong MongoDB, socket rooms cho realtime, pick problem theo difficulty, timer server-side, record submission verdict và finalize kết quả trận.

Socket.IO Redis sorted set MongoDB ELO update

Service communication

SubmissionService gọi ProblemService bằng x-service-token. EvaluationService gọi SubmissionService bằng x-service-token. BattleService gọi UserService để lấy username/ELO và update ELO, gọi SubmissionService để tạo submission.

HTTP internal Bearer JWT x-service-token

Tại sao tách các service này?

Service Vì sao tách riêng Cách thay thế Tradeoff
UserService Auth là boundary nhạy cảm: password, token, email verification, ELO user. Để auth chung trong monolith hoặc dùng Auth0/Clerk/Supabase Auth. Tự build auth giúp học sâu nhưng phải chịu trách nhiệm security; managed auth giảm rủi ro nhưng ít control.
ProblemService Problem/testcase là domain riêng; judge endpoint cần bảo vệ hidden testcase. Gộp problem vào SubmissionService hoặc quản lý qua CMS/admin backend. Tách riêng rõ ownership, nhưng thêm service-token và network call khi submit.
SubmissionService Là source of truth của submission state, lịch sử và accepted stats. Worker ghi trực tiếp DB submission hoặc gộp worker vào service. Callback qua API rõ boundary hơn, nhưng thêm service-to-service failure mode.
EvaluationService Chạy code untrusted là tác vụ nguy hiểm/nặng; nên cô lập khỏi API service. Chạy Docker ngay trong SubmissionService request handler. Tách worker giúp scale/cô lập, nhưng deploy phức tạp hơn và cần queue.
BattleService Realtime socket, matchmaking queue và timer khác hẳn CRUD API. Polling từ frontend hoặc để Battle trong monolith. Socket cho UX tốt, nhưng scale nhiều instance khó hơn HTTP stateless.

05 · Data model

Model dữ liệu và lý do thiết kế

Dự án dùng polyglot persistence: PostgreSQL cho identity/token có constraint mạnh, MongoDB cho domain documents có cấu trúc linh hoạt.

PostgreSQL/Prisma

  • User: username/email unique, password hash, role, profileVisibility, eloRating, emailVerified, createdAt.
  • RefreshToken: tokenHash, expiresAt, revokedAt, replacedByTokenId. Hỗ trợ refresh token rotation và revoke.
  • EmailToken: tokenHash, purpose, payload, expiresAt, usedAt. Payload dùng cho password change để lưu password hash mới.

MongoDB/Mongoose

  • Problem: title, description, difficulty, category, tags, isForBattle, editorial, testcases.
  • Submission: userId, problemId, code, language, isPracticeRun, status, verdict, testcase results, judgeMeta.
  • BattleRoom: mode, status, difficulty, roomCode, ownerId, problem, players, timer fields, winnerUserId.

Đánh giá lựa chọn DB hiện tại

Domain DB hiện tại Đánh giá Tradeoff Nếu làm lại / scale
User/Auth PostgreSQL + Prisma Rất hợp lý User có unique constraint, token relation, transaction refresh rotation. SQL xử lý tốt consistency và constraint. Giữ PostgreSQL. Có thể thêm audit table, login attempt table, session management.
Problem MongoDB Hợp lý cho MVP Problem là document gồm description/editorial/testcases/tags, đọc nguyên cụm nhiều hơn join. Mongo dễ lưu testcase array. Nếu cần admin workflow/versioning/testcase lớn, SQL cũng tốt: Problem, TestCase, Editorial tables. Mongo vẫn ổn nếu problem document không quá lớn.
Submission MongoDB Khá hợp lý Submission là event/log document: code, verdict, judgeMeta, testcaseResults. Append-heavy và query theo user/problem/status. Mongo hợp. Nếu muốn analytics mạnh, có thể stream sang ClickHouse/Postgres read model.
BattleRoom MongoDB Chấp nhận được, nhưng cần hiểu tradeoff Room state là document tự nhiên: players array, problem snapshot, timer, result. Update nguyên room đơn giản. Nhưng battle có quan hệ user-room, lịch sử, ELO transaction; SQL sẽ enforce tốt hơn. Với scale/production, cân nhắc PostgreSQL cho BattleRoom, BattlePlayer, BattleSubmissionEvent. Redis vẫn giữ queue/realtime cache.
Ranked queue Redis sorted set Rất hợp lý Queue cần thao tác nhanh theo score ELO, không cần lưu vĩnh viễn. Redis sorted set đúng bài toán. Giữ Redis, thêm distributed lock cho matcher nếu nhiều instance.

Mongo cho Battle có sai không?

Vì sao Mongo vẫn dùng được

Một battle room trong app hiện tại luôn chỉ có 2 players, vòng đời ngắn, state thường được đọc/ghi nguyên room. Document model giúp lưu room như một aggregate: mode, status, players, problem snapshot, timer và result nằm chung. Với quy mô này, Mongo không “sai”; nó giúp code nhanh và schema linh hoạt.

Vì sao SQL có thể tốt hơn

Battle là domain có quan hệ rõ: Room có nhiều Player, Player liên kết User, Submission event liên kết Room/User/Submission. Nếu cần audit, ranking chính xác, transaction kết thúc trận + update ELO + ghi ledger, SQL dễ đảm bảo consistency hơn.

Khuyến nghị thực tế: Giữ User/Auth ở PostgreSQL là đúng. Submission có thể giữ Mongo. Problem tùy định hướng: nếu problem/testcase/editorial cần versioning và admin nghiêm túc thì chuyển SQL sẽ rõ hơn; nếu ưu tiên document linh hoạt thì Mongo vẫn hợp. Battle nếu muốn production hóa nên cân nhắc chuyển sang SQL hoặc ít nhất tách thêm BattlePlayer/BattleEvent để query/history/audit tốt hơn.
Entity Index/constraint đáng chú ý Mục đích
User username unique, email unique Ngăn trùng danh tính; phục vụ login và public handle.
RefreshToken userId, tokenHash Tìm token để rotate/revoke, phát hiện reuse/mismatch.
EmailToken userId, tokenHash Xác minh link email một lần, có expiry và usedAt.

06 · Use cases and flows

Luồng nghiệp vụ quan trọng

Các flow bên dưới là phần bạn nên nắm thật chắc để trả lời phỏng vấn: request đi qua service nào, state nào được update, và tại sao thiết kế như vậy.

Use case 1: Đăng ký và xác minh email

1
User gửi username/email/password.

UserService validate Zod, hash password bằng bcrypt và tạo User chưa verified.

2
Tạo EmailToken SIGNUP_CONFIRM.

Raw token gửi qua email; database chỉ lưu SHA-256 hash, expiry và usedAt.

3
User click link.

Frontend route /auth/confirm?token=... gọi API confirm. Token được consume một lần.

4
Login chỉ cho email verified.

Nếu chưa verified, UserService trả Unauthorized.

Use case 2: Practice run và submit

1
Frontend tạo submission.

/submissions/run-samples cho Run, /submissions cho Submit. Cả hai cần Bearer access token.

2
SubmissionService fetch judge problem.

Gọi ProblemService /problems/:id/judge bằng service token để lấy full testcase.

3
Lưu submission và enqueue.

Submission ban đầu có status QUEUED. Run sample đánh dấu isPracticeRun=true và chỉ lấy 2 testcase đầu.

4
Evaluation worker xử lý.

Worker set RUNNING, chạy Docker theo từng testcase, tính verdict/runtime, callback status.

5
Frontend polling.

UI poll submission đến khi FINISHED hoặc INTERNAL_ERROR, sau đó refresh history và accepted stats.

Use case 3: Ranked matchmaking

1
User bấm tìm trận.

Socket emit ranked:enqueue. BattleService kiểm tra user chưa ở room active rồi thêm user vào Redis sorted set với score là ELO.

2
Match loop tìm cặp.

Service duyệt Redis queue theo ELO và chỉ match hai user cùng difficulty range: <1000 easy, 1000-1400 medium, >1400 hard.

3
Tạo room ACTIVE.

Chọn random battle problem cùng difficulty, set timer, lưu players với username/ELO trước trận.

4
Notify realtime.

Socket join hai user vào room, emit battle:match-found, frontend navigate qua battle room.

Use case 4: Private room

1
Owner tạo room.

BattleService tạo room WAITING, sinh roomCode 6 ký tự, enforce mỗi user chỉ ở một room active.

2
Challenger join bằng ID.

Room chỉ joinable nếu WAITING/READY và active players chưa đủ 2. Người thứ ba bị từ chối.

3
Owner start.

Room chuyển ACTIVE, pick problem, set endsAt, socket emit battle:room-started.

4
Cancel/leave trước start.

Owner cancel room; challenger leave room. Frontend điều hướng về battle home và current-state sync tránh stuck.

Use case 5: Kết thúc battle

1
Submit trong battle.

BattleRoomPage emit room:submit. BattleService tạo submission qua SubmissionService bằng access token của user.

2
Record verdict.

BattleService polling submission, record last verdict và bestRuntimeMs nếu AC.

3
Finalize.

Hết giờ hoặc cả hai leave. Nếu cả hai không AC thì hòa. Nếu một người AC thì người đó thắng. Nếu cả hai AC thì runtime thấp hơn thắng.

4
ELO.

Chỉ ranked match update ELO: winner +30, loser -25, draw 0. Private match không cộng/trừ ELO.

08 · Docker judge deep dive

Docker chạy code như thế nào?

Đây là phần quan trọng nhất của dự án nếu nhà tuyển dụng hỏi về judge system. Hãy hiểu theo pipeline: HTTP request chỉ tạo submission; worker phía sau mới thật sự chạy code.

Core backend concept

Flow từ lúc bấm Submit đến lúc có verdict

1
Frontend gửi code.

Practice dùng POST /submissions cho full submit hoặc POST /submissions/run-samples cho sample run. Request kèm Bearer access token. Battle submit đi qua socket room:submit, rồi BattleService gọi SubmissionService bằng access token của user.

2
SubmissionService xác thực và lấy testcase.

Service validate user, problemId, language, code. Sau đó gọi ProblemService /problems/:id/judge bằng x-service-token để lấy full testcase. Lý do: frontend/public API không được biết hidden testcase.

3
Lưu submission trạng thái QUEUED.

Submission được lưu vào MongoDB với code, language, userId, problemId, status=QUEUED, verdict=null. Với sample run, isPracticeRun=true và problem chỉ giữ 2 testcase đầu.

4
Đẩy job vào BullMQ.

SubmissionService enqueue job gồm submissionId, problem judge payload, code và language vào Redis/BullMQ. HTTP response trả submission ngay, không chờ Docker chạy.

5
EvaluationService worker nhận job.

Worker consume queue, gọi SubmissionService update status sang RUNNING, rồi chạy từng testcase tuần tự bằng runCode().

6
Tạo Docker container cho testcase.

Với mỗi testcase, code và input được ghi vào file trong container bằng heredoc. Container chạy command compile/run tương ứng với Python hoặc C++.

7
Đọc logs và tính verdict.

Runner đọc stdout/stderr, bỏ marker nội bộ, normalize output và so sánh với expected output. Nếu sai output là WA, timeout là TLE, lỗi compile là CE, lỗi runtime là RE.

8
Callback kết quả.

Worker gọi PATCH /submissions/:id/status bằng service token để set FINISHED, verdict, testcaseResults và judgeMeta. Frontend polling thấy status mới và render result.

Docker container thật sự chạy lệnh gì?

Python

Container ghi code.py và input.txt. Sau đó chạy python3 -m py_compile code.py để bắt syntax error trước, rồi chạy python3 code.py < input.txt. Runtime hiển thị được đo quanh phase chạy file Python.

C++

Container ghi code.cpp và input.txt. Sau đó chạy g++ code.cpp -o run. Nếu compile thành công, runner chạy ./run < input.txt. Runtime hiển thị chỉ đo phase chạy binary, không tính compile. 

SubmissionService
  ├─ save Submission(status=QUEUED)
  └─ BullMQ.add("evaluate-submission", { submissionId, problem, code, language })

EvaluationService worker
  ├─ update Submission(status=RUNNING)
  ├─ for each testcase:
  │    ├─ docker.createContainer(image, command, limits)
  │    ├─ container.start()
  │    ├─ wait for exit or kill after timeout
  │    ├─ read logs
  │    ├─ extract __JUDGE_RUNTIME_MS__
  │    └─ compare normalized actual vs expected
  └─ update Submission(status=FINISHED, verdict, judgeMeta)

Resource limit hiện tại

Limit Giá trị hiện tại Ý nghĩa Tradeoff
Timeout 4000ms/testcase Sau 4 giây, runner kill container và trả TLE. Dễ hiểu và an toàn, nhưng mỗi testcase có timeout riêng nên bài nhiều testcase có tổng thời gian lớn hơn.
Memory 1GB/container Docker HostConfig Memory giới hạn RAM container. Rộng cho local, nhưng quá nặng với free-tier 1GB; demo nên giảm 256-512MB.
CPU 50% của 1 CPU core CpuQuota=50000, CpuPeriod=100000. Giảm abuse CPU nhưng runtime sẽ chậm hơn máy thật; cần nhất quán khi so runtime battle.
PIDs 100 Giới hạn số process/thread trong container. Ngăn fork bomb cơ bản, nhưng production nên harden thêm.
Network none Code trong container không gọi internet/network. Đúng cho judge; nếu bài yêu cầu network thì không hỗ trợ.
Privilege no-new-privileges Ngăn privilege escalation trong container. Tốt, nhưng vẫn nên thêm seccomp/apparmor/user namespace/read-only FS.

Vì sao chọn Docker + queue?

Phương án Lợi ích Vấn đề Nhận xét
Chạy code trực tiếp trên host Nhanh, đơn giản, ít overhead. Cực nguy hiểm vì user code có thể đọc file, gọi network, chiếm CPU/RAM. Không nên dùng cho code untrusted.
Docker per testcase như hiện tại Cô lập tốt hơn, dễ setup language image, cleanup sau mỗi testcase. Overhead tạo container lớn, tốn tài nguyên, free-tier khó gánh. Hợp cho demo và correctness; cần tối ưu nếu scale.
Docker per submission Giảm overhead vì nhiều testcase chạy trong một container. Phức tạp hơn: cần test harness gom input/output, timeout per testcase khó hơn. Là hướng nâng cấp tốt nhất cho performance.
Sandbox chuyên dụng như isolate/firecracker/gVisor An toàn và production-grade hơn. Setup khó hơn Docker thường, cần Linux/kernel/runtime hiểu sâu. Hướng tốt nếu muốn judge nghiêm túc.
Câu trả lời phỏng vấn gọn: “Tôi không chạy code trong request handler. Submission được lưu và enqueue, worker riêng chạy code trong Docker. Docker cho isolation, BullMQ cho async/retry/scale worker. Hiện tại mỗi testcase tạo container riêng để đơn giản và an toàn, tradeoff là overhead lớn; nếu production tôi sẽ chuyển sang chạy nhiều testcase trong một sandbox hoặc dùng isolate/gVisor.”

09 · Socket.IO deep dive

Socket trong dự án hoạt động ra sao?

Socket.IO được dùng cho battle vì battle cần server đẩy event ngay lập tức: match found, room update, timer, opponent submit, verdict và result.

Realtime domain

Socket khác HTTP ở điểm nào?

HTTP request/response

Client hỏi, server trả lời. Muốn biết có update mới thì client phải polling. Hợp cho CRUD: list problem, get profile, create submission.

Socket realtime

Client giữ kết nối mở. Server có thể chủ động emit event cho user/room. Hợp cho battle vì đối thủ submit/leave hoặc timer tick cần hiển thị ngay.

Handshake và xác thực

1
Frontend tạo socket.

Client gọi io(BATTLE_SOCKET_URL, { auth: { token }, transports: ["websocket"] }).

2
BattleService middleware verify JWT.

Socket middleware đọc token từ handshake auth hoặc Authorization header, verify bằng JWT_ACCESS_SECRET, yêu cầu payload có userId và type access.

3
Lưu user vào socket.data.

Sau khi verify, server lưu { userId, accessToken } vào socket.data.user. Access token này dùng khi BattleService tạo submission thay user.

4
Register socket id.

Server lưu userSockets: Map<userId, Set<socketId>> để một user có thể mở nhiều tab mà vẫn nhận event.

Socket room là gì?

Socket.IO room là một nhóm socket id trong memory của Socket.IO server. Khi server gọi io.to(roomId).emit(...), tất cả socket trong room đó nhận event. Trong dự án, mỗi battle room MongoDB có room.id; socket room cũng dùng chính id này. 

User A tab 1 ─┐
User A tab 2 ─┼── join Socket.IO room "battleRoomId" ── receives timer/result/opponent events
User B tab 1 ─┘

emitToRoom(roomId, "battle:room-updated", room)
emitToRoomExceptUser(roomId, userA, "battle:opponent-submitted", payload)
emitToUser(userA, "battle:verdict", payload)

Ranked queue bằng socket

1
User bấm Find match.

Client emit ranked:enqueue. Server rate-limit event, kiểm tra user chưa ở room active.

2
Redis sorted set.

Server lấy ELO user từ UserService rồi zadd battle:ranked:queue elo userId. Score là ELO nên queue có thứ tự.

3
Matcher loop.

Cứ mỗi MATCHMAKING_INTERVAL_MS, server duyệt queue và tìm hai user cùng ELO range/difficulty. Khi match, server zrem cả hai khỏi queue.

4
Tạo ACTIVE room.

Server chọn battle problem theo difficulty, tạo Mongo BattleRoom, join sockets của hai user vào room và emit battle:match-found.

Private room bằng socket

Event Ai emit Server làm gì Vì sao dùng socket
private:create Owner Tạo room WAITING, sinh roomCode, join owner socket vào room. Owner nhận room ngay, sau đó chờ realtime update khi challenger join.
private:join Challenger Validate room joinable, thêm player, nếu đủ 2 thì READY, emit room-updated. Owner thấy challenger join ngay, không cần refresh.
private:start Owner Pick problem, set ACTIVE, set endsAt, emit room-started. Cả hai client chuyển sang battle room đồng thời.
private:cancel/private:leave Owner/challenger Cập nhật room CANCELED hoặc player hasLeft, emit event. Client còn lại bị sync về battle home đúng trạng thái.

Timer và battle result

Timer không chạy ở frontend như source of truth. Frontend chỉ hiển thị. Server mới là nơi giữ endsAt, set interval mỗi giây emit battle:timer, và khi hết giờ gọi finalizeRoom(). Cách này tránh mỗi client tự tính khác nhau hoặc user chỉnh clock local.

Vì sao không dùng polling?

Phương án Lợi ích Chi phí Áp dụng trong dự án
Polling HTTP Dễ deploy, stateless, load balancer đơn giản. Trễ event, nhiều request rỗng, timer/opponent action kém mượt. Đang dùng cho submission status practice, vì polling 1 submission là chấp nhận được.
Socket.IO Realtime, server push event, room broadcast dễ. Cần quản lý connection, reconnect, memory maps, scale nhiều instance khó hơn. Dùng cho battle vì domain cần realtime và broadcast giữa hai user.
Server-Sent Events Đơn giản hơn socket cho server-to-client stream. Một chiều; client vẫn cần HTTP POST riêng cho actions; room broadcast ít tiện hơn. Có thể dùng cho timer/event feed, nhưng Socket.IO tiện hơn cho bidirectional battle.
Tradeoff socket hiện tại: BattleService dùng in-memory userSockets, userCurrentRoom và roomTimers. Điều này ổn cho single instance/local demo. Nếu chạy nhiều instance, một user ở instance A có thể không nhận event từ instance B. Production cần Socket.IO Redis adapter, sticky sessions hoặc event bus, và timer nên có distributed lock.

10 · Security review

Bảo mật hiện tại và rủi ro

Dự án đã có nhiều yếu tố bảo mật đúng hướng, nhưng vẫn còn các hardening cần làm trước production.

Điểm mạnh

  • Password được hash bằng bcrypt, không lưu plaintext.
  • Access token và refresh token tách secret, có type claim để tránh dùng nhầm token.
  • Refresh token được hash trong DB và rotate khi refresh; token reuse/mismatch revoke toàn bộ refresh token của user.
  • Email token chỉ lưu hash, có purpose, expiry và usedAt, dùng được một lần.
  • Service-to-service endpoint quan trọng dùng x-service-token.
  • Problem public detail không expose full testcases; judge endpoint require service token.
  • Markdown được sanitize khi tạo/cập nhật problem để giảm XSS từ description/editorial.
  • Docker runner có allowlist ngôn ngữ, timeout 4s/testcase, memory 1GB, PIDs limit 100, CPU quota 50% một core, network none và no-new-privileges.
  • Socket.IO CORS đã restrict theo FRONTEND_ORIGIN thay vì wildcard.
  • Problem mutation routes yêu cầu admin JWT; judge/batch/difficulty internal routes yêu cầu service token.
  • Các action dễ abuse đã có rate limit cơ bản: auth/password, submission create/run và socket queue/submit.

Rủi ro cần nêu rõ

  • Frontend lưu token trong localStorage; nếu XSS xảy ra thì token có thể bị đọc. Production nên cân nhắc httpOnly secure cookie hoặc CSP nghiêm ngặt.
  • Service token là shared static secret; production nên rotate, scope theo service hoặc dùng mTLS/service mesh.
  • Rate limit hiện là in-memory, phù hợp local/single instance; production nhiều instance nên chuyển sang Redis-backed rate limiter.
  • Docker sandbox đã có CPU/PID/network/no-new-privileges nhưng vẫn nên thêm read-only FS, seccomp/apparmor profile, user namespace và image hardening.
  • Problem admin auth phụ thuộc JWT role ADMIN; cần admin UI/seed role và audit log nếu production hóa.
Nói trong phỏng vấn: “Project hiện tập trung chứng minh architecture và flow. Nếu production hóa, tôi sẽ ưu tiên API gateway, auth hardening, rate limit, container sandbox hardening, centralized observability và secret management.”

Khả năng scale

Dự án đã có nền tảng scale tốt ở phần judge nhờ queue, nhưng realtime battle và API topology cần thêm cơ chế phân tán nếu chạy nhiều instance.

Scale tốt hiện tại

Submission evaluation tách khỏi request lifecycle. Có thể tăng số EvaluationService worker để xử lý queue nhanh hơn. BullMQ retry job với backoff khi worker lỗi.

Điểm nghẽn

Mỗi testcase tạo một container riêng và chạy tuần tự. Với problem nhiều testcase hoặc nhiều user, overhead Docker sẽ lớn. Có thể batch testcase trong một container hoặc pool container.

Realtime scale

Socket.IO hiện dùng in-memory maps cho user sockets, current room và room timers. Nếu scale nhiều BattleService instance cần Redis adapter và distributed timer/lock.

Vì sao không làm monolith?

Monolith Express Dễ deploy, ít env, debug đơn giản, transaction dễ hơn. Judge Docker, auth, battle realtime và problem CRUD cùng process; lỗi judge có thể ảnh hưởng toàn app. Phù hợp MVP rất nhỏ hoặc team ít người cần ship nhanh.

Câu hỏi phỏng vấn khả năng cao

Vì sao dùng queue cho judge?

Vì chạy code là tác vụ nặng, lâu và không nên block HTTP request. Queue giúp request trả nhanh, worker scale độc lập, retry khi lỗi, và kiểm soát concurrency tốt hơn.

Vì sao tách SubmissionService và EvaluationService?

SubmissionService là source of truth của submission lifecycle; EvaluationService là compute worker. Tách ra giúp scale worker theo tải judge và cô lập rủi ro khi chạy code untrusted.

Runtime được đo như thế nào?

Command trong container ghi marker runtime bằng timestamp nanosecond quanh giai đoạn chạy chương trình, sau compile. Nhờ vậy runtime không bị tính cả thời gian create/start container hoặc compile.

Ranked matchmaking dựa trên gì?

User được đưa vào Redis sorted set với score ELO. Matcher lấy queue theo ELO và chỉ ghép hai user cùng difficulty range. Difficulty cũng quyết định problem pool và timer mặc định.

Private match khác ranked thế nào?

Private match có owner/challenger, room code, owner start/cancel. Kết quả vẫn tính win/loss/draw nhưng không update ELO. Ranked thì room ACTIVE ngay khi match và ELO thay đổi khi kết thúc.

Nếu production hóa bạn làm gì đầu tiên?

Thêm API gateway, rate limiting, secure cookie/CSP, CORS restriction, sandbox hardening, centralized logs/tracing, Socket.IO Redis adapter, distributed locks và integration/E2E tests.

Mongo cho Battle có ổn không?

Với room 2 người, vòng đời ngắn và đọc/ghi theo aggregate room thì Mongo dùng được. Nhưng nếu cần transaction/audit/ranking ledger chặt, tôi sẽ chuyển BattleRoom/BattlePlayer/BattleEvent sang PostgreSQL.

Vì sao battle dùng socket còn practice dùng polling?

Practice polling một submission là đơn giản và đủ tốt. Battle cần server push cho event của đối thủ, timer và result, nên socket giúp UX realtime hơn và giảm polling rỗng liên tục.
